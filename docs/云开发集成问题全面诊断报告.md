## 二、根本原因分析

### 问题 1：cloud.js 的 getOpenid() 实现错误

**文件**: `utils/cloud.js:58-60`
**当前代码**:
```javascript
function getOpenid() {
  return wx.getStorageSync('openid') || ''
}
```

**问题分析**:
```
从本地存储读取 openid，但：
1. 小程序从未设置过 openid 到本地存储
2. openid 应该从微信登录后的用户信息中获取
3. 云开发 API 有自己的内置方法来获取 openid
```

**微信云开发正确的 openid 获取方式**:
```javascript
// 方式 1：使用云函数获取（最安全）
const res = await wx.cloud.callFunction({
  name: 'getOpenid'
})
const openid = res.result.openid

// 方式 2：通过云调用上下文（仅在云函数中）
const openid = this.userInfo.openId

// 方式 3：通过认证的用户信息（云数据库会自动使用）
// 云数据库默认使用登录态的 openid，不需要手动传递
```

## 四、关键发现

### 4.1 云数据库权限模型的真实工作机制

```
权限设置："仅创建者可读写"

对每条记录来说：
├─ 创建时
│  ├─ 自动设置 _openid = 当前登录用户的 openid
│  └─ 其他用户无法访问此记录
├─ 读取时
│  ├─ 系统自动过滤，只返回 _openid === 当前用户的记录
│  └─ 所以即使你查询 where({ followingId: 'xxx' })
│     也只会返回当前用户的记录
└─ 删除时
   └─ 系统自动只删除 _openid === 当前用户的记录

❌ 现在的问题：
   getOpenid() 返回 ''，所以 _openid = ''
   结果所有人的关注都混在一起了
```

### 4.2 为什么说"数据库没有更新"

实际上数据可能已经保存了，但：

```
假设：
User A 点击关注 User B
openid('') → 被保存为 _openid = ''

后来 User C 也点击关注
openid('') → 也被保存为 _openid = ''

当 User A 查询自己的关注时：
SELECT * FROM follows WHERE _openid = 'User A'
结果：无记录！（因为记录的 _openid = ''）

当 User A 查询所有数据时：
显示的是 _openid = '' 的记录，看起来很混乱
```

## 五、修复方案

### 5.1 修复 cloud.js

```javascript
// 正确的实现方式
function getOpenid() {
  // 方案 1：从登录态获取（最可靠）
  try {
    const userInfo = wx.getUserInfo?.({ withCredentials: true })
    return userInfo?.openid || ''
  } catch (e) {
    console.log('[cloud] 获取 openid 失败')
  }
  
  // 方案 2：备用：从本地存储（用户登录时保存）
  return wx.getStorageSync('openid') || ''
}
```

**但最根本的问题**：
云数据库不需要你手动传递 openid！系统会自动添加。

修复后的 followService：
```javascript
async function saveCloudFollow(followingId, followingData = {}) {
  const { db } = cloudModule
  // 不要手动设置 openid！
  
  const data = {
    followingId,
    followingData,
    createTime: db.serverDate()
    // _openid 由系统自动设置
  }
  
  await db.collection('follows').add({ data })
  return true
}
```
